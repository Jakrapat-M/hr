Purpose

The HR project runs two Claude sessions in parallel: /linear-loop (plan-only) and /linear-implement (ZTE build + self-merge). Throughput is good, but integrity is not: merged PRs sometimes replace features that sibling tickets already shipped, worktrees get mixed (edits and untracked files leaking between tickets and the shared checkout), and the BA reports that Vercel does not reflect the implementation after a ticket is marked done. This plan closes all three gaps with deterministic tooling — a single serialized merge gate, a worktree isolation guard, a hot-file collision policy, and a build-identity check that proves what Vercel is actually serving — plus the doc updates that make the two loops obey them.

Problem

Three symptoms, five root causes. All of them are consequences of running parallel writers against one master with no single enforced landing lane.

#Root causeSymptom it produces
R1 Stale-base merge. A PR branched days ago stays MERGEABLE on GitHub (no textual conflict) even after siblings merged. Merging it lands its old copy of shared files — GitHub's mergeability check knows nothing about semantic reverts. "The code always conflicts / replaces the feature we already built."
R2 Last-merged-wins on hot files. Nearly every ticket touches messages/en.json, messages/th.json, sidebar nav, registries, and mock-data seeds. Two open PRs both pass CI independently; whichever merges second overrides the first. Shipped feature's i18n keys / menu entry / registry row silently vanish.
R3 Multiple merge paths. Merges happen via ad-hoc gh pr merge, merge-ready.sh --go (mergeability-only, no rebase-validation), ship.mjs merge (has a stale-base guard), and the GitHub UI. Only pr-merge-gate.sh validates on the tip the PR will actually land on — and nothing forces it to be used. Gate exists but reverts still ship; inconsistent safety per merge.
R4 Worktree contamination. Sessions edit the shared checkout (its git status is currently full of mixed untracked files), git stash is shared across all worktrees, worktrees get reused across tickets or left behind, and nothing verifies a worktree's branch was cut from fetched origin/master. "Some mix worktree" — cross-ticket files land in the wrong PR; branch switches clobber uncommitted work.
R5 No deploy-truth check. After merge, nobody verifies the production Vercel deployment rebuilt and is serving the merge commit. Linear comments link preview URLs that go stale when the branch is deleted, or localhost screenshots. And when R1/R2 reverted the feature, Vercel is faithfully showing a master that genuinely lost it. "BA says Vercel is not reflecting the real implementation when done."
Why the existing tools didn't stop this: git-hygiene.sh only warns; merge-ready.sh merges anything GitHub calls MERGEABLE; pr-merge-gate.sh does the right validation (rebase onto fresh master, feature-preservation sentinel, baseline-diffed tests) but is optional, only checks en.json, and treats PR-vs-PR overlap as a warning, not a block. Enforcement — not detection — is the missing piece.

Solution

Keep the two-session parallelism for authoring, but make landing strictly serial and verified. Four invariants, each enforced by a script (not by prompt discipline), then encoded into the loop docs:

  • [x] I1 — One merge lane: the only way code reaches master is pr-merge-gate.sh --go, which rebases the PR onto fresh origin/master, re-runs build + baseline-diffed tests, and asserts no existing i18n key / route disappeared — then merges serially, re-gating each next PR on the new tip. ship.mjs merge and merge-ready.sh both delegate to it.
  • [x] I2 — One ticket, one worktree, one session: a worktree-guard.sh preflight refuses implementation edits in the shared checkout, verifies branch ↔ ticket ↔ worktree mapping, verifies the branch contains fetched origin/master, blocks on cross-worktree file overlap, and registers every worktree in .omc/state/worktrees.json so leftovers are visible and pruned.
  • [x] I3 — Hot files merge serially: overlap between two open PRs on the known conflict magnets (en.json/th.json, sidebar nav, registries, mock-data seeds) upgrades from warning to gate failure — merge one, force-rebase the other, re-gate. Locale parity is checked for both languages, not just EN.
  • [x] I4 — Vercel is proven, not assumed: the app exposes /api/version (commit SHA + env). After every merge, a postmerge step polls production until it serves the merge SHA, and only then posts the BA-facing Linear comment — with the production URL, the build SHA, and screenshots taken from that verified URL.
Design principle: every fix is a deterministic script change or a doc change that points at the script. No fix relies on an agent "remembering" a rule — that is exactly what has been failing.

Notes

Why serialization, not smarter merging

The tempting fix is a custom JSON merge driver for en.json/th.json so concurrent edits auto-merge. Rejected: merge drivers don't run on GitHub's server-side merges, they mask semantic conflicts (two tickets renaming the same key), and they add a per-machine .gitattributes/config dependency. Serializing hot-file landings through the gate is slower per merge but is deterministic everywhere and costs ~2 minutes per rebase.

Assumptions baked into this plan

  • BA looks at the production Vercel URL (master deployments), not per-PR previews. If it turns out the BA reviews previews, Phase 4's poller targets stay the same — the immutable per-commit preview URL rule already covers that side.
  • The Vercel project deploys master → production automatically. The concrete prod URL is resolved during implementation and committed to .claude/skills/ship/config.json.
  • GitHub branch rulesets may not be enforceable on this private repo's plan — that task (1.3) is best-effort by design; the gate script is the real enforcement.
  • No new npm dependencies anywhere: the version route uses only process.env; the poller uses Node's built-in fetch; guard/gate are plain bash + python3 (already required by the existing scripts).
