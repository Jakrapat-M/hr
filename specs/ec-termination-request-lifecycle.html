Purpose

Make the termination request a single end-to-end demo flow: an employee (ESS) or a manager/HRBP submits a termination request with the full SF-parity field set, the approver sees every submitted field and can approve or send back, and the requester tracks it in My Requests — where the info shown matches exactly what was submitted, and revise/withdraw become available only after a send-back. All four tickets share the same surfaces and the same store, so they ship as one plan (mockup phase: Zustand-persisted state, no real backend).

Problem

Today the lifecycle is fragmented across two disconnected implementations and a lossy record type:

  • STA-269 — the admin terminate form (/admin/employees/[id]/terminate) hides everything behind a two-step EffectiveDateGate: the user must confirm "Resigned Date" before the form even renders. June's mock wants the Resigned Date input inline on the same page, next to the auto-computed Termination date. The Personal Email field also lacks the required remark.
  • STA-266 — the ESS resignation form (/resignation) has only 5 fields (last working date, reason, personal email, additional info, attachments). It's missing the auto-computed Termination date and the sub-reason ("Reason for termination"), doesn't share the admin form's reason logic (TERMINATION_LOGIC), and its Personal Email has no remark.
  • STA-264 — the approval detail (/workflows/resignation/[id]) shows only reason + free text + attachments. The 10-field list June specified can't render because TerminationRequest (in termination-approvals.ts) never carried terminationDate, voluntary, reasonForTermination, transferOutTo, okToRehire, or a distinct additionalInfo. Worse, the admin form doesn't create a request at all — it commits the termination immediately to the employee timeline, so there is nothing for an approver to approve.
  • STA-271 — in /requests a resignation row only opens a generic approval-chain modal, not the submitted info. "Send back" is implemented as the store's reject(), which drops the row from the queue entirely — so the STA-193 withdraw/revise buttons (which key off sent-back status) can never appear for termination requests.

Solution

Converge both submit paths onto one enriched TerminationRequest record and one shared read-only renderer, then let the status machine carry the whole lifecycle:

  1. Enrich the record + status machine (Phase 1): add the missing SF fields to TerminationRequest, add a real sent_back status (distinct from rejected), add sendBack / updateRequest / withdraw actions, and unify reason codes on TERMINATION_LOGIC. Build one shared <TerminationRequestSummary> component that renders June's field list from a request — reused verbatim by the approval page (STA-264) and My Requests (STA-271), which guarantees "shown info same as UI when submit".
  2. Admin form inline date + queue routing (Phase 2, STA-269): drop the gate on this page, render Resigned Date + read-only Termination date side-by-side per the mock, add the Personal Email remark, and make submit create a TerminationRequest (approval happens in the queue; the timeline commit + status: terminated flip move to final approval).
  3. ESS field parity (Phase 3, STA-266): same field logic as the admin form, reduced UI — exactly the 7 fields June listed, with the email remark.
  4. Full-field approval + send-back (Phase 4, STA-264): approval detail renders the summary component (all 10 fields when present); actions are Approve and Send back (with note); send-back sets sent_back and returns the request to the submitter.
  5. My-Task parity + revise/withdraw (Phase 5, STA-271): the row's detail shows the same summary; sent_back projects as the "info" row status so the existing STA-193 withdraw / revise-and-resubmit UI activates — and only then. Revise reopens the original form pre-filled and resubmits the same request id back to pending_manager.

Notes

Decisions taken (and why)

Admin terminate submit now goes through the approval queue (biggest behavior change)

Today the admin form commits immediately (timeline + status:'terminated') and nothing reaches an approver. STA-264 ("manager/hr can approve or sendback") + STA-271 ("if manager/HR submit request: shown info same as UI when submit" in My Task) only make sense if the admin submission becomes a TerminationRequest. The commit moves to final approval. Confirm with June/BA whether an HR-admin submission should skip the manager stage (start at pending_spd) — plan assumes the existing 2-step chain for both roles; trivially adjustable in addRequest routing.

Send back ≠ reject

The current UI's "ส่งกลับ" maps to reject(), which drops the row — making STA-271's "revise and withdraw only after approval send back" impossible. A first-class sent_back status keeps the request alive and routes it to the requester. rejected is retained in the store for a future true-reject action but gets no primary UI this phase.

Reason-code sets unified on TERMINATION_LOGIC

The store enum (17 codes, incl. typo'd TERM_RETRIE) and the admin registry (TERM_RETIRE, TERM_DM, …) diverged. STA-266 says the ESS form's logic must match the manager form, so TERMINATION_LOGIC_CODES becomes canonical; a mapper normalizes legacy persisted values in the v2 migration. This also gives the ESS form the sub-LOV ("Reason for termination") for free.

Date minimums intentionally differ per role

ESS keeps the 30-day-notice minimum (existing policy help text + notice check on the approval page); the admin form keeps min = tomorrow. June's field lists don't ask to change either. The auto-compute rule (termination = resigned + 1 day) is identical on both, via the shared helper.

"Attachment ID" interpreted as the attachment upload/list, not a text input

SF's field is literally "Attachment ID", but both existing forms already implement attachments as AttachmentDropzone and the mock shows a dropzone. The summary renderer lists file names — no internal-artifact ID input is exposed (mockup-data caveat in CLAUDE.md).
