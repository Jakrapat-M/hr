Executive Overview · สำหรับพี่เคน นำเสนอ HR / Business

หนึ่งแพลตฟอร์ม HR สำหรับพนักงาน 70,000+ คน — ทดแทน SuccessFactors ด้วย Self-Service ที่เร็วกว่า ปลอดภัยกว่า และครอบคลุมกว่า

เอกสารฉบับนี้สรุปข้อกำหนดผลิตภัณฑ์ (Product Requirements) ของ Central Retail Next-Gen HRMS ครอบคลุม 29 module ใน 6 domain group สำหรับ 4 persona พร้อม cross-module journey และเส้นทางอนุมัติ (workflow) ทุกประเภท โดยอ้างอิงเฉพาะหลักฐานที่มีในตัว repository — PRD, spec module, route map ของ frontend, และไฟล์แปลภาษา.

ครอบคลุม 29 modules เป้า ลดงาน HR Admin 60% FCP < 1.5s · ปัจจุบัน 3–4s รองรับ Thai/English (i18n) RBAC 4 ระดับ + Audit Trail

Scope · ปัจจุบัน

Phase ปัจจุบัน = UI Mockup สมบูรณ์ ก่อน backend

อ้างอิงจาก CLAUDE.md · 2026-05-15

  • In scope: clickable flows, modals, tables, filters, navigation, state transitions, realistic seed data, Humi design conformance, TH/EN parity.
  • Out of scope: real API/POST/PUT, optimistic-vs-pessimistic logic, retry/error UX beyond basic, observability, RBAC enforcement vs real IAM.
  • Definition of done: HR Team สามารถ step ผ่านทุก admin/employee flow ได้โดยไม่มี dead end และ sign-off วิสัยทัศน์ UI + interaction.
Carveout · STA-25

Configuration-Based Plan Setup (Benefit Administrator)

Linear ticket STA-25 · overrides "no internal artifacts" rule สำหรับ 9-tab configurator

  • Admin persona "Benefit Administrator" ต้องกรอก TTT, template, schemaVersion เป็น user-input fields ใน Identity tab ของ configurator.
  • กฎ "ห้ามแสดง internal artifacts" ยังคงใช้กับ CreatePlanModal / EditPlanModal เดิม (PRs #148 / #149) เท่านั้น.
  • HR ต้องช่วยตัดสินใจว่า persona "Benefit Administrator" ใช้สิทธิ์เดียวกับ HR Manager หรือเป็น role ใหม่.
Open Decision

ขอบเขต Auto-Approve & Delegation

สังเกตจาก settings.md — feature ยังเป็น "อยู่ระหว่างการพัฒนา"

  • Threshold สำหรับ auto-approve leave / OT / claims ยังไม่มีตัวเลขที่ตกลงร่วมกัน (ลาเกินกี่วัน, OT กี่ชั่วโมง, claim ไม่เกินกี่บาท).
  • Delegation chain (จำนวนวันสูงสุด, สิทธิ์การมอบต่อ) ยังต้องการ HR policy ก่อน implement.
  • Bulk approve — เพดาน 50 รายการต่อ batch กำลังเป็น Could Have ใน F8.2 อาจต้องปรับเป็น Must.
Open Decision

Hi-Po Criteria & Calibration Cadence

สังเกตจาก talent-management & succession-planning

  • เกณฑ์ Hi-Po (Performance ≥ X, Potential ≥ Y, อายุงานขั้นต่ำ, อายุสูงสุด) ยังไม่ fix.
  • รอบ Calibration Review Session (ทุกไตรมาส / รายปี) ต้องเลือกร่วมกับ HR Director.
  • Drag-and-Drop ระหว่างช่อง 9-Box ยังต้องตัดสินว่าจะมี audit log หรือไม่.
Backend-later boundary

โซนที่ยังเป็น Mock Data

ใช้ realistic seeds (e.g., BENEFIT_PLAN_REGISTRY, mock claim history) เพื่อให้ HR ตัดสินใจ visual + interaction direction ก่อน

  • Workflow engine: routing rules & persistence ยังเป็น in-memory mock.
  • Payroll processing — variance / exception detection ใช้ data generator.
  • Smart Claims OCR: ผลลัพธ์ที่แสดงเป็น mock confidence + extracted fields.
  • Auth: ไม่มี real IAM — ใช้ role switch ใน UI สำหรับ demo.
Compliance · TH

Thai Labor Law & Compliance Boundary

PRD ระบุ "PIT, SSO, PDPA สมบูรณ์" แต่หลายเงื่อนไขยังรอ legal review

  • OT limit 36 ชั่วโมง/สัปดาห์ — บังคับใช้ใน overtime เรียบร้อย แต่ exception handling สำหรับ shift work ยังไม่ครอบคลุม.
  • PDPA: data masking, retention period, "right to be forgotten" workflow ยังไม่มี spec.
  • 50 ทวิ ออกได้เฉพาะ PDF — confirm legal-acceptable signing method.
