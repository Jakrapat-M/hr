# บท 10 — Git และการส่งงาน

> เป้าหมาย: บันทึกงานอย่างปลอดภัย (ย้อนได้เสมอ) และส่งงานขึ้นให้ทีมรีวิว
> Git คือ "เซฟเกม" ของโค้ด — พังได้ ย้อนได้เสมอ

---

## 1. แนวคิด Git ใน 1 นาที

- **commit** = เซฟสแน็ปช็อตของงาน ณ จุดหนึ่ง (พร้อมข้อความอธิบาย)
- **branch (กิ่ง)** = สายงานแยก ทำของใหม่โดยไม่กระทบของหลัก
- **master** = กิ่งหลัก (ของที่ใช้จริง) — **อย่าแก้บนนี้ตรงๆ**
- **push** = ส่งงานขึ้น GitHub
- **PR (Pull Request)** = ขอเอางานของเรารวมเข้า master (ให้คนรีวิวก่อน)

> เปรียบเทียบ: master = ต้นฉบับหนังสือ / branch = ฉบับร่างที่เราขีดเขียน
> เขียนเสร็จค่อยขอรวมกลับ (PR) ให้บรรณาธิการตรวจ

---

## 2. ⭐ กฎเหล็กของโปรเจกต์นี้ (สำคัญมาก)

1. **แตกกิ่งจาก master เสมอ** — อย่าแก้บน master ตรงๆ อย่าต่อกิ่งซ้อนกิ่งที่ยังไม่ merge
2. **commit เมื่อพร้อม/เมื่อถูกขอ** — ไม่ commit มั่วๆ กลางคัน
3. **ทำทีละเรื่อง** — 1 PR = 1 เรื่อง อย่ารวมหลายเรื่องในก้อนเดียว
4. **คนเป็นคนกด merge** — AI/เราไม่ auto-merge เอง ให้เจ้าของโปรเจกต์กด
5. **อย่า stage ไฟล์ noise** — `logs/*.json` (ถูก gitignore แล้ว) และ `.omc/state/*`
   (ตัวนี้ **ถูก track ใน git** จะโผล่ใน `git status` เป็น Modified ตลอด) เป็นไฟล์ที่ระบบเขียนเอง
   → อย่า `add` มันไปกับงานเรา ใช้ `git add <ไฟล์ของเราเอง>` ทีละไฟล์เท่านั้น

## 3. ขั้นตอนมาตรฐาน (ทำตามนี้ทุกครั้ง)

### ขั้น 0 — เริ่มจากที่สะอาด
```bash
cd ~/Projects/hr
git checkout master        # ไปอยู่บนกิ่งหลัก
git pull                   # ดึงของล่าสุดลงมา
git status                 # ดูว่าตอนนี้มีอะไรค้างไหม
```

### ขั้น 1 — แตกกิ่งใหม่ (ตั้งชื่อสื่อความ)
```bash
git checkout -b fix/profile-duplicate-key
#              ↑ ชื่อกิ่ง: ประเภท/เรื่องสั้นๆ เช่น fix/... หรือ feat/...
```

### ขั้น 2 — แก้โค้ด (บท 08) + ทดสอบ (บท 09)
แก้ → ดูจอ → `npm run build` ผ่าน

### ขั้น 3 — เลือกเฉพาะไฟล์ที่เราตั้งใจแก้ (อย่าเหมารวม noise)
```bash
git status                                    # ดูว่าไฟล์ไหนเปลี่ยนบ้าง
git add src/frontend/src/app/[locale]/profile/me/page.tsx   # เลือกทีละไฟล์ที่เราแก้จริง
```
> 🚫 อย่าใช้ `git add -A` หรือ `git add .` ลวกๆ — มันจะลากไฟล์ noise (`logs/*`, `.omc/state/*`) ติดไปด้วย
> เลือกเฉพาะไฟล์ที่เราตั้งใจแก้

### ขั้น 4 — commit พร้อมข้อความอธิบาย
```bash
git commit -m "fix(profile): unique React keys in employment FieldCard"
```
รูปแบบข้อความที่ดี: `ประเภท(ขอบเขต): อธิบายสั้นว่าทำอะไร`
- ประเภท: `fix` (แก้บั๊ก), `feat` (ฟีเจอร์ใหม่), `docs` (เอกสาร), `refactor` (จัดระเบียบ)
- เขียนเป็นภาษาอังกฤษสั้นๆ บอก "ทำอะไร" ไม่ใช่ "แก้ไฟล์ไหน"

### ขั้น 5 — push ขึ้น GitHub
```bash
git push -u origin fix/profile-duplicate-key
```

### ขั้น 6 — เปิด PR
```bash
gh pr create --base master --title "fix(profile): unique React keys" --body "อธิบายว่าแก้อะไร ทำไม พิสูจน์ยังไง"
```
หรือเปิด GitHub บนเบราว์เซอร์แล้วกดปุ่ม "Compare & pull request"

### ขั้น 7 — รอรีวิว + คนกด merge
อย่ากด merge เอง รอเจ้าของโปรเจกต์ตรวจและกด

---

## 4. ปุ่มย้อนกลับ (เครื่องมือกู้ชีพ)

| สถานการณ์ | คำสั่ง |
|-----------|--------|
| แก้ไฟล์มั่วไปหมด อยากคืนค่าเดิม (ไฟล์เดียว) | `git restore <ไฟล์>` (แบบเก่า: `git checkout -- <ไฟล์>`) |
| อยากดูว่าแก้อะไรไปบ้าง | `git diff` |
| ลบ commit ล่าสุด แต่เก็บการแก้ไว้ | `git reset --soft HEAD~1` |
| อยากกลับไป master ทิ้งกิ่งนี้ | `git checkout master` |

> 💡 ตราบใดที่ commit แล้ว งานไม่หาย — Git จำได้หมด สบายใจได้

---

## 5. เรื่องเฉพาะระบบนี้

- มีไฟล์ noise ที่เปลี่ยนเองตลอด (`logs/*.json`, `.omc/state/*`) — **ปล่อยไว้ ไม่ต้อง commit**
- บาง branch ของฟีเจอร์อาจยัง merge ไม่เสร็จ — **รอมัน merge เข้า master ก่อน** แล้วค่อยแตกกิ่งใหม่จาก master (อย่าต่อยอดบนกิ่งที่ยังลอย)
- **Linear (ระบบติดตามงาน):** ก่อนเริ่มฟีเจอร์/แก้บั๊กที่มาจากงานวางแผน ให้ค้น Linear ว่ามี ticket อยู่ไหม
  อ่าน description + เงื่อนไขความสำเร็จ (AC) ก่อน แล้วอ้าง ticket id ใน commit + PR
  - ถ้าไม่มี ticket → บอกให้ชัดใน PR ว่า "ไม่มี ticket"
  - **สถานะ `Done` เลื่อนโดยคนเท่านั้น** — เราเลื่อนได้สูงสุดแค่ `In Review` (กันปิดงานก่อนตรวจจริง)

---

## สรุปบทนี้

ลำดับทอง: `master` → `pull` → แตกกิ่ง → แก้ → build ผ่าน → `add` เฉพาะไฟล์ → `commit` → `push` → เปิด PR → รอคน merge

กฎ: แตกกิ่งจาก master / ทีละเรื่อง / เลือกไฟล์เอง ไม่เหมา noise / ไม่ merge เอง
