# บท 09 — ทดสอบและตรวจสอบ (พิสูจน์ว่าใช้ได้จริง)

> เป้าหมาย: ก่อนบอกว่า "เสร็จแล้ว" ต้องมีหลักฐาน ไม่ใช่แค่ "น่าจะได้"
> กฎทองของโปรเจกต์นี้: **เห็นผลจริงบนจอ > เทสต์เขียว**

---

## 1. ด่านตรวจ 3 ระดับ (จากเบาไปหนัก)

| ด่าน | คำสั่ง | ตรวจอะไร | ใช้ตอน |
|------|--------|----------|--------|
| 1. Build | `npm run build` | โค้ดไม่มี error, type ถูก | **ทุกครั้งก่อนส่งงาน** |
| 2. Unit test | `npm test` | logic ย่อยๆ ถูกต้อง | เมื่อแตะ logic |
| 3. ดูจอจริง | เปิดเบราว์เซอร์ / Playwright | หน้าจอแสดงถูกจริง | **เสมอ** |

---

## 2. ด่าน 1: `npm run build` (ด่านสำคัญสุด)

```bash
cd ~/Projects/hr/src/frontend
npm run build
```
- ใช้เวลา 1-3 นาที
- จบด้วย **exit 0** = ผ่าน (ไม่มี error, type ถูกหมด)
- ถ้ามี error จะขึ้นชื่อไฟล์ + บรรทัด → ไปแก้ตรงนั้น

> นี่คือด่านเดียวกับที่ "type check" — ถ้า build ผ่าน แปลว่าชนิดข้อมูลถูกหมด
> **อย่าส่งงานถ้า build ไม่ผ่าน** เด็ดขาด

---

## 3. ด่าน 2: Unit test (Vitest)

```bash
npm test                          # รันทั้งหมด
npm test -- --run <คำค้น>          # รันเฉพาะที่เกี่ยวข้อง (เร็วกว่า)
```
ตัวอย่าง: `npm test -- --run payroll` รันเฉพาะเทสต์ที่ชื่อมี "payroll"

เทสต์เก็บไว้ 2 ที่:
- `src/__tests__/` (เทสต์รวม)
- ข้างไฟล์ที่มันเทส เช่น `xxx.test.tsx` (เทสต์เฉพาะที่)

> ระบบนี้มีเทสต์เยอะ (2,300+ เคส) ส่วนใหญ่เขียว
> ถ้าแก้แล้วมีเทสต์แดง → อ่านว่ามันคาดหวังอะไร แล้วตัดสินว่า "โค้ดเราผิด" หรือ "เทสต์ต้องอัปเดต"

### วิธีอ่านผลเทสต์แดง (อย่าเพิ่งตกใจ)

ผลเทสต์ที่ fail จะหน้าตาประมาณนี้:
```
 FAIL  src/__tests__/payroll-summary.test.tsx > shows team total
 AssertionError: expected '฿828,194.00' to be '฿820,000.00'
   ❯ src/__tests__/payroll-summary.test.tsx:42
   42|   expect(total).toBe('฿820,000.00')
```
อ่านจากบนลงล่าง:
1. **บรรทัด FAIL** → ไฟล์ไหน เทสต์ชื่ออะไร (`shows team total`)
2. **expected ... to be ...** → "ได้ค่า X แต่คาดหวัง Y" (เห็นความต่างชัดๆ)
3. **ไฟล์:บรรทัด** (`...test.tsx:42`) → เปิดไปดูว่าเทสต์เช็คอะไรตรงนั้น

จากนั้นถาม: "ค่าที่ได้จริง (X) ถูกหรือผิด?"
- ถ้า X **ถูก** (เราตั้งใจเปลี่ยนพฤติกรรม) → อัปเดตเทสต์ให้คาดหวัง X
- ถ้า X **ผิด** (เราทำพังโดยไม่ตั้งใจ) → กลับไปแก้โค้ดให้ได้ Y

> 💡 รัน `npm test -- --run <ชื่อไฟล์>` เพื่อรันเฉพาะตัวที่แดง ซ้ำๆ ระหว่างแก้ (เร็วกว่ารันทั้งหมด)

## 4. ด่าน 3: ดูจอจริง (สำคัญที่สุด — กฎทองของที่นี่)

> เคยมีเคสจริง: เทสต์เขียวหมด แต่หน้าจอจริงแสดงวันที่เป็น "-" (บั๊ก)
> → **เทสต์เขียวไม่พอ ต้องเปิดดูจอจริงด้วยตาเสมอ**

วิธีที่ 1 — **ดูด้วยตา** (ง่ายสุด): เปิดเบราว์เซอร์หน้าที่แก้ กดใช้งานจริง
ที่ต้องเช็ค:
- หน้าแสดงครบไหม (ไม่มีช่องว่าง/ข้อความหาย)
- กดปุ่ม/ฟอร์มทำงานไหม
- เปิด Console (F12) มี error สีแดงไหม
- ลองทั้ง `/th` และ `/en`

วิธีที่ 2 — **Playwright** (อัตโนมัติ เปิดเบราว์เซอร์จริงให้): ใช้ตอนอยากทดสอบหลาย persona/หน้าเร็วๆ
มีสคริปต์ตัวอย่างใน repo และเทสต์ใน `e2e/*.spec.ts` รันด้วย:
```bash
npm run test:e2e -- --project=chromium <ชื่อ spec>
```

> สำหรับเคน: **วิธีที่ 1 (ดูด้วยตา) พอแล้วสำหรับงานทั่วไป** Playwright ไว้ทำตอนคล่องขึ้น

## 6. ตรวจสไตล์โค้ด (lint) — เสริม

```bash
npm run lint
```
> ⚠️ หมายเหตุสำคัญ: lint ตั้งให้ตรวจ "เฉพาะ ~9 ไฟล์ในลิสต์" (ดู `scripts.lint` ใน `package.json`)
> **ไม่ใช่ทั้งโปรเจกต์** — ถ้าเคนแก้ไฟล์ที่ไม่อยู่ในลิสต์ `npm run lint` จะ "เขียวผ่าน" แม้มีปัญหา
> (อย่าหลงเชื่อว่า lint เขียว = ปลอดภัยทั้งหมด) ด่านจริงที่ครอบทุกไฟล์คือ `npm run build`

## 7. เช็กลิสต์ "เสร็จจริง"

ก่อนพูดว่าเสร็จ ตอบให้ได้ทุกข้อ:
- [ ] เปิดดูจอจริงแล้ว มันถูกจริง (มีหลักฐาน/สกรีนช็อต)
- [ ] Console ไม่มี error/warning ใหม่
- [ ] `npm run build` = exit 0
- [ ] ถ้าแตะ logic → `npm test` ที่เกี่ยวข้องเขียว
- [ ] ครบสองภาษา (ถ้าแตะข้อความ)
- [ ] ไม่มีสีแดง / ไม่มีสีดิบ (บท 11)

> ❗ ถ้าด่านไหนไม่ผ่าน = ยังไม่เสร็จ กลับไปแก้ ไม่ใช่ "ปล่อยผ่าน"
